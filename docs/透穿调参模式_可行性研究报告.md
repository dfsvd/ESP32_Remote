## 1. 需求背景

### 痛点

飞场调参无电脑可用。飞控通常通过 USB 连接电脑使用 Betaflight Configurator 调参，但飞场环境不具备此条件。

### 用户理想流程

```
手机打开 app.betaflight.com → 通过 RC 手柄中转 → 连接飞控 → 完成调参
```

### 核心要求

1. **无线调参**：不依赖 USB 物理连接
2. **手机端**：直接用浏览器打开 Betaflight Web Configurator，不装额外 App
3. **与桌面版体验一致**：调 PID、滤波、端口、CLI 等全功能可用
4. **调参同时可操控飞机**：摇杆→CRSF→飞控的链路在调参时保持连接
5. **BLE 手柄功能不需要**：模拟器用的 BLE HID 在透穿模式下不启用

## 2. 可行性分析

### 2.1 ESP32 USB 硬件限制

原始 ESP32 的 USB 控制器为 OTG 规格，但存在以下限制：

| 因素 | 说明 | 影响 |
|------|------|------|
| **USB 角色** | 同一时间只能是 Host 或 Device | 当前 TinyUSB 占 Device 模式（HID 手柄），无法同时做 Host |
| **OTG 切换** | 可切 Host 但需停用 TinyUSB 并释放驱动 | 会丢失所有 USB HID 功能 |
| **PHY 引脚** | OTG 需 GPIO 18/19 | 与当前硬件设计不兼容 |
| **VBUS 供电** | ESP32 GPIO 无法直接为飞控 VBUS 供电 | 需外部 5V 升压电路 |

**结论**：ESP32 通过 USB 直连飞控的方案不可行。

### 2.2 CRSF MSP 透穿方案 ✅（采用）

#### 现有链路分析

```
飞控 ←[CRSF UART]→ 接收机(RX) ←[ExpressLRS RF]→ 高频头(TX) ←[CRSF UART]→ ESP32(手柄)
```

这条链路**已经存在**，高频头通过 UART (GPIO 16/17, 400kbps) 连接 ESP32，飞控和接收机之间也通过 CRSF 协议通信。

#### CRSF 协议中的 MSP 支持

CRSF 协议原生定义了 MSP 透穿帧类型：

| 帧类型 | 值 | 说明 |
|--------|-----|------|
| `CRSF_FRAMETYPE_MSP_REQ` | `0x7A` | MSP 请求（从遥控端发往飞控） |
| `CRSF_FRAMETYPE_MSP_RESP` | `0x7B` | MSP 响应（从飞控发往遥控端） |

**ExpressLRS 高频头在这两个帧类型上的行为**：

- TX 模块收到 `0x7A` 帧后，将 MSP payload 通过 RF 链路转发给 RX 模块
- RX 模块将 MSP payload 以 `0x7A` 帧格式发送给飞控
- 飞控处理 MSP 命令后发送 `0x7B` 响应帧
- RX 模块通过 RF 链路转发回 TX 模块
- TX 模块以 `0x7B` 帧发送给 ESP32

**来源**：ExpressLRS 源码中的 CRSF 协议实现，帧类型 0x7A/0x7B 为保留的 MSP 透传通道。

#### 可行性结论

✅ **无需任何硬件改动**，CRSF MSP 透穿方案完全可行。MSP 命令在 CRSF 帧中透明传输，飞控端无感知。

### 2.3 Web BLE 兼容性

#### Betaflight Configurator Web 版连接方式

通过分析 Betaflight Configurator 源码（v2025.12+），确认 Web 版支持以下连接方式：

| 连接方式 | 源码文件 | 平台 |
|----------|---------|------|
| **Web BLE** | `src/js/protocols/WebBluetooth.js` | 桌面 Chrome |
| **Capacitor BLE** | `src/js/protocols/CapacitorBle.js` | Android 原生 |
| Web Serial | `src/js/protocols/WebSerial.js` | 桌面 Chrome |
| WebSocket/TCP | `src/js/protocols/WebSocket.js` | 调试用 |

#### 支持的 BLE 设备列表

来源：`src/js/protocols/devices.js`

| 设备名 | Service UUID | Write Char | Read/Notify Char |
|--------|-------------|-----------|-----------------|
| CC2541 | `0000ffe0-...` | `0000ffe1-...` | `0000ffe2-...` |
| **HM-11** | **`6e400001-b5a3-f393-e0a9-e50e24dcca9e`** | **`6e400003-b5a3-f393-e0a9-e50e24dcca9e`** | **`6e400002-b5a3-f393-e0a9-e50e24dcca9e`** |
| **Nordic NRF** | **`6e400001-b5a3-f393-e0a9-e50e24dcca9e`** | **`6e400003-b5a3-f393-e0a9-e50e24dcca9e`** | **`6e400002-b5a3-f393-e0a9-e50e24dcca9e`** |
| SpeedyBee V1 | `00001000-...` | `00001001-...` | `00001002-...` |
| SpeedyBee V2 | `0000abf0-...` | `0000abf1-...` | `0000abf2-...` |
| DroneBridge | `0000db32-...` | `0000db33-...` | `0000db34-...` |

**关键发现**：Nordic UART Service（NUS）的 UUID `6e400001-...` 已被 Betaflight Configurator 原生支持，Web BLE 连接时会自动匹配为 "HM-11" 或 "Nordic NRF" 设备类型。

#### Web BLE 连接流程（源码分析）

来自 `WebBluetooth.js` 的 `connect()` 方法：

```
1. navigator.bluetooth.requestDevice()  // 用户选择设备
2. device.gatt.connect()                // 连接 GATT
3. server.getPrimaryServices()          // 枚举服务
4. 匹配 service.uuid 到 knownDevices    // 识别为 HM-11 / Nordic NRF
5. service.getCharacteristics()         // 获取读写特征
6. characteristic.startNotifications()  // 启动 notify
7. writeCharacteristic.writeValue()     // 发送 MSP 数据
8. handleNotification()                 // 接收 MSP 响应
```

#### 结论

✅ **app.betaflight.com 完全可以通过 Web BLE 连接 ESP32 的 NUS 服务**。用户在 Chrome 中点 Connect → Bluetooth → 选择 "FPV RC BLE V2"，即可实现 MSP 透传。

### 3.2 数据流

```
MSP 命令发送（配置写入）：
  手机 BLE → [MSP字节] → ESP32 NUS RX → Bridge → 0x7A帧 → CRSF UART → 高频头TX
  → RF → 接收机RX → CRSF UART → 飞控

MSP 响应接收（配置读取）：
  飞控 → CRSF UART → 接收机RX → RF → 高频头TX → CRSF UART → ESP32
  → 0x7B帧解析 → Bridge → BLE NUS TX → 手机

RC 通道数据（摇杆控制，持续发送）：
  摇杆ADC → joy结构体 → CRSF 0x16帧 → CRSF UART → 高频头TX → RF
  → 接收机RX → 飞控
```

### 3.3 BLE 服务定义

采用 Nordic UART Service (NUS)，与 Betaflight Configurator 原生兼容：

| BLE 项目 | UUID | 属性 | 说明 |
|----------|------|------|------|
| **Service** | `6e400001-b5a3-f393-e0a9-e50e24dcca9e` | - | NUS 服务 |
| **TX Characteristic** | `6e400002-b5a3-f393-e0a9-e50e24dcca9e` | Notify | ESP32→手机，推送 MSP 响应 |
| **RX Characteristic** | `6e400003-b5a3-f393-e0a9-e50e24dcca9e` | Write Without Response | 手机→ESP32，接收 MSP 命令 |

广播数据：仅含 NUS Service UUID，设备名 "FPV RC BLE V2"。

### 3.4 CRSF MSP 帧格式

#### 0x7A - MSP_REQ (ESP32 → 飞控)

```
┌──────┬──────┬──────┬──────┬──────┬────────────────┬──────┐
│ Dest │ Len  │ 0x7A │ Src  │ FC   │ MSP Payload    │ CRC  │
│ 0xEE │ N+4  │      │ 0xEA │ Dest │ (N bytes)      │      │
└──────┴──────┴──────┴──────┴──────┴────────────────┴──────┘
```

#### 0x7B - MSP_RESP (飞控 → ESP32)

```
┌──────┬──────┬──────┬──────┬──────┬────────────────┬──────┐
│ Dest │ Len  │ 0x7B │ Src  │ FC   │ MSP Payload    │ CRC  │
│ 0xEA │ N+4  │      │ 0xEA │ Dest │ (N bytes)      │      │
└──────┴──────┴──────┴──────┴──────┴────────────────┴──────┘
```

MSP Payload 为**原始 MSP 数据帧**（无 CRSF 封装），由飞控原生解析。透穿层不修改 MSP 内容。

## 5. 实施步骤

### Step 1: CRSF MSP 帧支持

**文件**: `rc_crsf.c` / `rc_crsf.h`

#### 新增常量

```c
#define CRSF_FRAMETYPE_MSP_REQ  0x7A
#define CRSF_FRAMETYPE_MSP_RESP 0x7B
```

#### 新增 API

```c
/**
 * @brief 发送 MSP 数据到飞控（封装为 CRSF 0x7A 帧）
 * @param data 原始 MSP 字节
 * @param len  MSP 长度
 */
void crsf_send_msp(const uint8_t *data, size_t len);

/**
 * @brief 读取飞控返回的 MSP 响应
 * @param buf     接收缓冲
 * @param max_len 缓冲大小
 * @return 实际读取的字节数，0=无数据
 */
int crsf_read_msp(uint8_t *buf, size_t max_len);

/**
 * @brief 检查是否有 MSP 响应待读取
 */
bool crsf_msp_available(void);
```

#### 内部实现

**发送** (`crsf_send_msp`):
```c
void crsf_send_msp(const uint8_t *data, size_t len) {
    uint8_t frame[128];
    size_t total_len = 4 + len; // type(1) + sender(1) + receiver(1) + msp(N) + crc(1)
    frame[0] = s_target_addr;       // 0xEE
    frame[1] = total_len;
    frame[2] = CRSF_FRAMETYPE_MSP_REQ; // 0x7A
    frame[3] = s_target_addr;       // 0xEE (sender)
    frame[4] = CRSF_ADDRESS_RADIO;  // 0xEA (receiver → FC)
    memcpy(&frame[5], data, len);
    frame[5 + len] = crsf_crc8(&frame[2], 3 + len);
    crsf_uart_write(frame, total_len + 2); // +2 for dest + len
}
```

**接收**（在 `crsf_rx_task` 的 switch 中新增 case）:
```c
} else if (type == CRSF_FRAMETYPE_MSP_RESP) {
    // payload[0] = sender, payload[1] = receiver
    // payload[2..] = MSP data
    size_t msp_len = frame_len - 4; // total - type - sender - receiver - crc
    if (msp_len > 0 && msp_len <= 256) {
        // 写入 MSP 环形缓冲
        write_msp_ring_buffer(&payload[2], msp_len);
    }
}
```

**环形缓冲**: 使用 FreeRTOS StreamBuffer（或简单的循环缓冲），容量 2048 字节，足以容纳一次 MSP 响应（通常 < 512 字节）。

#### 条件编译与初始化

在 `ble_init()` 中根据是否透穿模式决定是否初始化 NUS：

```c
void ble_init(fpv_joystick_report_t *joy, bool enable_nus) {
```

### Step 3: 桥接任务

**新文件**: `rc_bridge.c` / `rc_bridge.h`

#### API

```c
/**
 * @brief 启动 BLE NUS ↔ CRSF MSP 桥接任务
 * @note  仅在透穿模式下调用
 */
void bridge_start(void);

/**
 * @brief 停止桥接任务
 */
void bridge_stop(void);
```

#### 实现

```c
#include "rc_crsf.h"
#include "rc_ble.h"
#include "freertos/FreeRTOS.h"
#include "freertos/task.h"

#define BRIDGE_TASK_STACK 4096
#define BRIDGE_TASK_PRIORITY 4
#define BRIDGE_POLL_MS 10

static TaskHandle_t s_bridge_task = NULL;

static void bridge_task(void *arg) {
    uint8_t buf[256];
    
    while (1) {
        // 方向 A: 手机 → 飞控
        int n = ble_uart_read(buf, sizeof(buf));
        if (n > 0) {
            crsf_send_msp(buf, (size_t)n);
        }
        
        // 方向 B: 飞控 → 手机
        if (crsf_msp_available()) {
            n = crsf_read_msp(buf, sizeof(buf));
            if (n > 0) {
                ble_uart_send(buf, (size_t)n);
            }
        }
        
        vTaskDelay(pdMS_TO_TICKS(BRIDGE_POLL_MS));
    }
}

void bridge_start(void) {
    if (s_bridge_task == NULL) {
        xTaskCreatePinnedToCore(
            bridge_task, "bridge", BRIDGE_TASK_STACK, NULL,
            BRIDGE_TASK_PRIORITY, &s_bridge_task, 1
        );
    }
}

void bridge_stop(void) {
    if (s_bridge_task != NULL) {
        vTaskDelete(s_bridge_task);
        s_bridge_task = NULL;
    }
}
```

## 7. 风险与注意事项

| 风险 | 说明 | 缓解措施 |
|------|------|---------|
| **CRSF 帧长限制** | CRSF 帧 payload 最大 120 字节（含帧头） | MSP 大包自动分片，Bridge 处理合包逻辑 |
| **RF 链路延迟** | 每次 MSP 往返约 20-100ms（取决于 ELRS 速率） | Betaflight MSP 协议天然容忍延迟，只影响读/写响应速度不丢数据 |
| **BLE MTU 分片** | MSP 响应可能超过 BLE MTU (默认 23 字节) | NUS TX 特征使用 Notify 属性，由 BLE 协议自动分片 |
| **同时连接限制** | BLE 只能同时连接一台手机 | 透穿模式不做 HID，全部带宽给 NUS |
| **GATT 表容量** | 当前 24 个定义槽位，HID 用 16 个，加 NUS 用 6 个 | 配置扩容至 32，有余量 |
| **MSP 协议版本** | Betaflight 不同版本 MSP 协议有差异 | Bridge 层不解析 MSP 内容，完全透明传输，无兼容性问题 |
| **掉线处理** | CRSF 链路断开时 MSP 请求会超时 | 桥接任务设置超时机制（如 3s 无响应返回空缓冲区），不给 BLE 端发死锁数据 |
